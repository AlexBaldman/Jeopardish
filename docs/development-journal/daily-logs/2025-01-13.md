### Phase 2: Game Logic Refactoring (90 min)
Extracted and reorganized game logic:
- `core/game.js` - State machine for game flow

- `core/scoring.js` - Score calculations
- `core/validation.js` - Answer validation with feedback
- `core/controller.js` - Thin orchestration layer

**Key Achievement**: Separated concerns completely. The controller knows nothing about scoring logic, scoring knows nothing about questions.

**Challenge**: The original code had game logic mixed with UI updates. Had to carefully extract the pure logic.

### Phase 3: State Management (90 min)
Built custom state management system:
- `state/store.js` - Central store with time travel
- `state/actions.js` - Atomic action definitions
- `state/selectors.js` - Memoized computed values
- `state/reducer.js` - Pure state transitions
- `state/persistence.js` - Auto-save functionality

**Why Not Redux?**: We built exactly what we needed - no more, no less. Our solution is 300 lines vs Redux's thousands.

**Breakthrough**: Time travel debugging will be invaluable for bug reports.

## Technical Decisions Made

1. **No Framework Initially**
   - Start with vanilla JS
   - Understand the actual needs
   - Can migrate to React/Vue later if needed

2. **Event-Driven Architecture**  
   - Components communicate via events
   - No direct coupling
   - Easy to test and debug

3. **Immutable State**
   - All state changes create new objects
   - Enables time travel
   - Prevents subtle bugs

4. **Functional Core, Imperative Shell**
   - Business logic is pure functions
   - Side effects at the edges
   - Maximum testability

## Performance Considerations

1. **Memoized Selectors** - Prevent redundant calculations
2. **Debounced Persistence** - Don't thrash localStorage
3. **Event Delegation** - Minimize event listeners
4. **Lazy Loading Ready** - Architecture supports code splitting
